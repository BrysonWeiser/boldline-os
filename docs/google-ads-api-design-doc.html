1. Purpose and business model

BoldLine Media is a digital marketing agency that manages Google Ads campaigns on behalf of small-business clients. Every client owns and pays for their own Google Ads account; BoldLine never handles client ad spend. Client accounts are linked to BoldLine’s manager account (MCC GOOGLE-ADS-MANAGER-ID-WITH-DASHES), through which the agency holds manager-level access only. This ownership model is stated publicly on our website (https://boldlinemedia.com).

The Google Ads API is used by BoldLine OS, our internal, in-house operations platform. The tool is used exclusively by BoldLine Media staff. It is not distributed, licensed, resold, or exposed to clients or any third party. Clients receive performance reports generated by the platform (email summaries and a read-only status page), but no client or external user ever operates the API tool or triggers API calls.

2. System architecture

BoldLine OS is a web application composed of a browser front end and server-side serverless functions (Node.js on Netlify Functions), with application data stored in Supabase (managed PostgreSQL). All Google Ads API traffic originates from the server-side functions; the browser never calls the Google Ads API directly.

Owner (agency staff) — authenticated session | v BoldLine OS web app (browser) | HTTPS (owner-authenticated requests) v Netlify serverless functions (Node.js) -----> Google Ads API (REST) | - developer token v - OAuth 2.0 refresh token Supabase (PostgreSQL) - login-customer-id: MCC (cached metrics, approval queue, report history)

2.1 Authentication and credential handling

  • OAuth 2.0 authorization-code flow, consented once by the MCC administrator; the resulting refresh token is exchanged server-side for access tokens per request batch.
  • The developer token, OAuth client credentials, and refresh token are stored as server-side environment variables in the hosting platform’s encrypted configuration. They are never present in client-side code, never committed to source control, and never shown in the UI.
  • All API calls set login-customer-id to the MCC and operate only on client accounts explicitly linked to the MCC.
  • Every serverless endpoint that can reach the Google Ads API first validates the owner’s authenticated session; unauthenticated requests are rejected.

3. Google Ads API usage

CapabilityAPI service / methodTypeWhat it does in BoldLine OS
Account connectivity check CustomerService.listAccessibleCustomers Read Verifies credentials and enumerates accounts linked to the MCC (used by an owner-facing “test connection” card).
Performance reporting GoogleAdsService.search (GAQL) Read Retrieves campaign-level metrics (impressions, clicks, cost, conversions; trailing 30 days) per linked client account to power the owner dashboard and scheduled client performance reports.
Budget adjustment CampaignBudgetService.mutateCampaignBudgets Write Updates a campaign budget — executed only after explicit human approval (section 4).
Campaign status change CampaignService.mutateCampaigns Write Pauses or re-enables a campaign — executed only after explicit human approval (section 4).

No other API services are called. The tool does not create accounts, does not manage billing, does not upload conversions, and does not use the App Conversion Tracking and Remarketing API.

4. Human-in-the-loop control for all write operations

No automated process ever changes a client’s ad account without prior human approval. Any proposed change (e.g. a budget adjustment or a campaign pause) is written to an in-app approval queue as a descriptive proposal. The agency owner reviews each proposal in the BoldLine OS interface and explicitly approves or rejects it. Only an approved proposal is executed via the mutate services above, and every decision is logged. Rejected proposals are never executed.

Reads (reporting queries) run on a schedule and on demand; writes occur only through the approval flow described above.

5. Expected API volume

  • Small number of linked client accounts (single digits at launch, growing gradually).
  • Reporting reads: scheduled refreshes a few times per day per account, plus occasional on-demand dashboard loads.
  • Writes: infrequent — typically a few approved changes per week across all accounts.
  • Estimated total: well under 1,000 operations per day — comfortably within Basic Access limits.

6. Data handling, storage, and privacy

  • Campaign metrics retrieved from the API are cached in Supabase (PostgreSQL) solely to render dashboards and generate client reports; row-level security restricts access to the authenticated owner.
  • Google Ads data is used only to service the account it came from. It is never sold, shared with third parties, combined across clients, or used for any purpose other than reporting to that client and managing that client’s campaigns.
  • All transport is HTTPS/TLS. No Google Ads credentials or tokens are ever stored in the database or exposed to the browser.

7. Policy compliance

  • The tool is internal-only; Required Minimum Functionality (RMF) obligations for third-party tools do not apply, and no API functionality is offered to external users.
  • BoldLine Media agrees to comply with the Google Ads API Terms and Conditions and related policies.
  • The API contact email (theboldlinemedia@gmail.com) is actively monitored.
