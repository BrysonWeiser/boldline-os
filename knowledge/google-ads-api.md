**✅ Approve→execute BUILT 2026-07-19** (the final wiring). How it works end to end:
- **ARIA sees real campaigns:** opening ARIA fetches `campaigns` for every client with `googleAdsCustomerId` (max 8, silent on failure) and injects a `LIVE GOOGLE ADS CAMPAIGNS` block into the system prompt (id, name, status, $X/day, 30-day clicks/conv/cost).
- **Executable proposals:** `propose_action` (aria.js) gained an optional `exec` payload — `kind` (`set_daily_budget` | `pause_campaign` | `enable_campaign`) + `campaign_id` + `campaign_name` (+ `new_daily_budget_dollars`, DAILY not monthly). The prompt tells ARIA to attach it only for campaigns in the live block, never invented ids. The OS intake validates exec (known kind, numeric id, budget 0<x<100k, client actually linked) and silently degrades to descriptive-only if malformed.
- **Approve = execute:** in Notifications, ⚡-chipped proposals execute on Approve: the OS **re-reads `campaigns` fresh** (verifies the campaign still exists + gets current `budgetResourceName`/`campaignResourceName` — never trusts proposal-time data), then calls `setBudget`/`setStatus` via the shared `gadsCall` helper. Success removes the card + commLog "Approved + EXECUTED in Google Ads: … (summary)". Failure keeps the card in the queue, shows the error inline, Approve becomes "Retry Execution". Reject never executes anything. Non-exec proposals (creative/targeting/Meta/unlinked clients) behave exactly as before — logged only.
- **Human-in-the-loop preserved:** nothing fires without the physical Approve tap — same guardrail story given to Google in the Basic Access application.
